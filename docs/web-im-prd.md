# Web IM 业务交互 PRD

> 版本 v0.1 · 2026-05-21 · 分支 `feature/auto-ops`
> 作用域:为 SL 多 Bot 网关补一个独立的 Web IM 前端,让用户可以脱离飞书直接在浏览器里调度运营团队(运营经理 + KOL增长 / SEO内容 / 社媒分发 三个子 Agent)。

---

## 1. 背景与目标

### 1.1 现状
SL 当前唯一入口是飞书:用户在飞书私聊或群聊里 @ 某个 Bot,Gateway 把消息转发给对应子进程,Worker 调 Claude 回复。痛点:
- 演示 / 评委 / 外部协作方没有飞书 → 无法体验。
- 飞书消息形态是"对话",看不到运营 SOP 的多步流程(找 KOL → 验邮箱 → 起草 → 审批 → 跟进)。
- 飞书侧无法做"人工审批"这种需要按钮 + 状态流转的交互。

### 1.2 目标(本期)
- **G1 · 浏览器代替飞书**:任意用户打开 Web 页面就能跟 4 个 Bot 对话,行为与飞书一致(私聊 + @群聊)。
- **G2 · 多 Agent 协作可观测**:看到"运营经理"如何把任务派给 KOL/SEO/分发,以及 Bot 之间 @ 转派。
- **G3 · 关键节点人工审批**:Bot 在执行高风险动作(发邮件 / 发布内容 / 报价)前推送审批卡片,人点"通过"才继续。

### 1.3 非目标(本期不做)
- 注册 / 多租户 / 权限管理(本期单实例,内网 / 局域网部署即可)。
- 富文本编辑器、文件存储 CDN、视频通话。
- 飞书侧的全部功能镜像(只对齐核心消息收发)。

## 2. 角色与典型场景

| 角色 | 描述 | 使用频次 |
|---|---|---|
| **运营负责人** | 在 Web IM 中下达"本周 KOL 触达 50 人"等任务,审批关键节点 | 每天数次 |
| **演示者(评委 / 客户)** | 观察整套 Auto-Ops Loop 怎么自动跑 | 一次性 |
| **(隐含)Bot** | 4 个 Agent 子进程,通过同一通道发消息 | 持续 |

### 典型用户故事

**US-1 · 运营负责人下任务**
> 我打开 Web IM → 进入"运营经理"私聊 → 输入"本周锁定 30 个 5K-50K 粉丝的 AI 工具垂类 KOL,周三前给我候选名单"。运营经理 @KOL增长 把任务派下去 → 我能看到群聊里 Bot 之间的对话进度 → KOL增长完成后把名单回贴到群里。

**US-2 · 审批卡片**
> KOL增长完成调研,准备群发 DM 之前 → 在群里推送审批卡(标题 + 收件人列表 + DM 文案预览 + 通过/驳回按钮)→ 我点"通过" → Bot 继续执行;我点"驳回 + 备注" → Bot 修改文案后再来一次。

**US-3 · 演示**
> 评委打开 Web IM → 默认进入"演示群"(预置 4 个 Bot + 一个引导脚本)→ 输入"演示一遍 KOL 触达完整流程" → 引导脚本驱动 4 个 Bot 协作 → 评委看到时间轴 + 消息流 + Agent 状态面板。

## 3. 信息架构

```
┌───────────────────────────────────────────────────────────────┐
│ 顶部栏: SL 控制台    · 在线 Bot:4/4    · 当前用户:游客 │
├──────────────────┬──────────────────────────┬─────────────────┤
│  会话列表(左)   │   消息区(中)            │ 详情面板(右)  │
│                  │                          │                 │
│ # 私聊            │ ┌─ 运营经理 ─────────┐  │ ▸ Agent 状态    │
│ ● 运营经理       │ │ 你好,本周需要 30  │  │   - 运营经理 ✅  │
│ ○ KOL增长        │ │ 个 KOL...           │  │   - KOL增长  ✅  │
│ ○ SEO内容        │ └─────────────────────┘  │   - SEO内容  ✅  │
│ ○ 社媒分发        │ ┌─ Me ───────────────┐  │   - 社媒分发 ✅  │
│                  │ │ ...                 │  │                 │
│ # 群聊            │ └─────────────────────┘  │ ▸ 当前任务      │
│ ◎ 运营战情室      │                          │   #T-2026-01    │
│ ◎ KOL 巡检日报   │ [审批卡片 ➜ 通过/驳回]   │   状态:审批中  │
│                  │                          │   负责:KOL增长 │
│ + 新建群聊       │ ┌ 输入框 + @ + 文件 + 发送│ ▸ 工具调用日志  │
└──────────────────┴──────────────────────────┴─────────────────┘
```

- **左栏 — 会话列表**:私聊(每个 Bot 一条) + 群聊(用户自建,可拉多个 Bot)。未读红点。
- **中栏 — 消息区**:气泡化(用户右、Bot 左)、Markdown 渲染、流式打字机效果、特殊卡片(审批/任务/工具调用)。
- **右栏 — 详情面板(可折叠)**:
  - **Agent 状态卡**:复用 `GET /bots`,展示 4 个 Bot 的 pid/status/restartCount。
  - **当前任务卡**:本会话关联的任务 ID + 状态 + 责任人。
  - **工具调用日志**:本次会话里 Bot 调过的工具(MCP tools 等),折叠展示。

## 4. 核心交互流程

### 4.1 收发消息

```
用户输入 → POST /web/message
              ↓
       Gateway 路由到目标 Bot 子进程(IPC)
              ↓
       Worker 调 Claude 流式输出
              ↓
       chunk → IPC → Gateway → WebSocket 推 → 浏览器逐字渲染
```

**对齐飞书行为的点**:
- 私聊:消息直接进入对应 Bot 的 ConversationStore。
- 群聊:必须 @ 某个 Bot 才触发(@all 触发全部);Bot 回复带 @用户。
- 同一 chatId 的会话历史复用 ConversationStore + 自动压缩,与飞书共用底层。

### 4.2 Bot 间 @ 转派

运营经理回复内容里包含 `@KOL增长 请去做 X` 时,前端解析 mention → 自动把这条消息也作为 input 送给 KOL增长(同 chatId 群聊),KOL增长流式回复。**无需用户介入**。

### 4.3 审批卡片(human-in-loop)

Bot 在 system prompt 里被告知"涉及发邮件 / 报价 / 公开发布前必须发起审批"。Bot 走特殊工具 `request_approval(payload)` → 网关把 payload 推到群里,渲染成卡片:

```
┌──── ⚠️ 待审批 · KOL增长 ────────┐
│ 类型:批量 DM 触达                │
│ 目标:20 位 KOL(展开列表)        │
│ 文案预览:"Hi {name}, 我们..."    │
│ 风险:外发内容 · 不可撤销         │
│ ─────────────────────────────── │
│ [ ✓ 通过 ]  [ ✗ 驳回(填备注) ] │
└──────────────────────────────────┘
```

- 通过 → 网关回 IPC `approval:approved` → Bot 继续。
- 驳回 → 备注作为新 user 消息塞回对话 → Bot 修改后再次发起。
- 5 分钟超时 → 自动驳回,提示"请人工再触发"。

### 4.4 演示模式

URL 携带 `?demo=kol-flow` → 进入预置群聊 + 引导脚本(用户只需点"开始演示")→ 系统按预定剧本依次注入 user 消息驱动 Bot 协作,前端把每个阶段标记到右侧时间轴。

## 6. 后端集成方案(推荐)

> 用户问过"复用 HttpServer / 独立服务 / 听建议"。**推荐:在现有主进程的 HttpServer 上扩 WebSocket + 新增一组 `/web/*` REST 路由**,理由:
> - 已有 IPC + Gateway + ConversationStore,直接复用。Bot 状态、消息收发、对话历史都不需要再搭一层。
> - 单进程拓扑简单,部署只多一个静态前端。
> - 飞书与 Web 共用同一份对话状态(同 chatId)→ 用户切换不丢上下文。

### 新增接口

```
GET    /web/bots                        # 与 /bots 一致,前端拉 Agent 列表
GET    /web/conversations/:botId        # 该 Bot 的会话列表
GET    /web/conversations/:chatId/messages?before=&limit=
POST   /web/conversations/:chatId/messages   # 发消息(返回 messageId,推流走 WS)
POST   /web/groups                      # 创建群聊(选 Bot 列表)
POST   /web/approvals/:id/decision      # 审批通过/驳回
WS     /web/stream                      # 订阅自己关心的 chatId,接收增量消息 / typing / 卡片 / 状态
```

### IPC 扩展

子进程 → 主进程新增:
- `WEB_TYPING_START / STOP`
- `WEB_APPROVAL_REQUEST(payload)`
- `WEB_TOOL_CALL_LOG(payload)`(复用现有 enableTools 链路)

主进程 → 子进程新增:
- `WEB_APPROVAL_DECISION(approved, note)`
- `WEB_USER_MESSAGE(chatId, content, mentions[])`

### 前端选型(建议但不强制)

| 维度 | 推荐 | 备注 |
|---|---|---|
| 框架 | React 18 + Vite | 上手快、生态好,主进程 HttpServer 直接 serve dist/ |
| 状态 | Zustand 或 Jotai | IM 场景轻量、避免 Redux 样板 |
| 通信 | 原生 WebSocket + fetch | 不引入 SSE/Socket.io |
| 渲染 | react-markdown + highlight.js | 代码块需要 |
| UI | shadcn/ui + Tailwind | 卡片 / 弹层一应俱全 |

## 7. 验收标准(本期)

| 编号 | 验收点 | 通过条件 |
|---|---|---|
| AC-1 | 私聊收发 | 选中"运营经理" → 输入消息 → 1s 内开始流式 → 完整回复落到 ConversationStore |
| AC-2 | 群聊 @ 触发 | 群里 @KOL增长 → 仅 KOL增长 触发回复;无 @ → 谁都不响应 |
| AC-3 | Bot 间转派 | 运营经理回复中带 `@KOL增长` → KOL增长在 5s 内开始输出 |
| AC-4 | 审批卡片 | Bot 走 `request_approval` → 卡片渲染 → 点通过 → Bot 收到 `approved` 事件继续 |
| AC-5 | 状态面板 | 4 个 Bot 全部显示 ✅;手动 SIGKILL 一个子进程 → 30s 内面板变红再变绿(自动重启) |
| AC-6 | 飞书共存 | 同一时间通过飞书发消息和 Web 发消息,二者历史相互可见(同 chatId) |
| AC-7 | 离线提示 | 网关重启 → 浏览器 WS 自动重连,期间消息进入"待重发"队列 |

## 8. 风险 & 待澄清(Open Questions)

| 编号 | 议题 | 影响 | 建议 |
|---|---|---|---|
| Q-1 | 是否需要登录 / 用户身份 | 决定能否做"个人收件箱"和审批权限 | 本期默认匿名,带 token query 区分 session;下一期再做账号 |
| Q-2 | 群聊里 Bot 之间是否能并行讲话 | 涉及消息顺序与并发锁 | 默认串行,@ 解析后逐个唤醒,后续按需放开 |
| Q-3 | 审批超时 5min 是否合理 | 影响夜间无人值守流程 | 让 Bot 在 system prompt 中决定是否标记"可异步等待" |
| Q-4 | 演示模式剧本格式 | 影响 F-11 实现复杂度 | 用 YAML 描述 step → user 消息,放在 `agents/{id}/demos/*.yaml` |
| Q-5 | 是否要导出审批 / 任务记录给飞书多维表格 | 业务闭环需要 | 列入 P1,本期只本地落 JSONL |
| Q-6 | 已存在 MCP Server,Web 是否再走 MCP 调工具 | 架构一致性 | 不走。Web 是"用户视图",MCP 是"AI 助手控制面",两者同源不同入口 |
